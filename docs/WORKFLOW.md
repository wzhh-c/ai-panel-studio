# 开发过程思路 & 工作流说明

## 一、开发流程概述

本次开发严格遵循 **SDD → DDD → TDD → E2E** 四阶段工程化范式，全程使用 Claude Code（Deepseek V4 Pro）作为核心开发工具。每阶段产出经人工质检确认后，再进入下一阶段。

### 阶段划分与关键产出

| 阶段 | 范式 | 关键产出 | AI 协作方式 |
| :--- | :--- | :--- | :--- |
| 准备规划 | 混合 | PRD、ER 图、API 契约 | 通过 Prompt 驱动 AI 生成草稿，人工评审定稿 |
| 后端开发 | SDD | Repository、Service、API、SSE | 定义 Schema 后逐层生成，每层独立测试 |
| 前端开发 | DDD | React 组件、状态管理、SSE 消费 | 先出 UI 原型，再添加交互逻辑 |
| 测试闭环 | TDD+E2E | 单元测试、集成测试、curl 验证 | 先写测试用例，再实现逻辑 |

## 二、AI 协同中的典型问题与解决路径

### 问题 1：`Transcript` 缺少 `sequence` 字段

**现象**：质检时发现 SQL 和 API 均缺少发言序号，可能导致同一秒内的发言顺序不确定。

**解决**：在阶段0 Prompt 中明确要求增加 `sequence` 字段，并在 `TranscriptRepository.create` 中实现自增逻辑（用 `SAVEPOINT` 保证原子性）。

**启示**：人工质检必须在 AI 产出后及时介入，否则问题会逐层放大。

---

### 问题 2：发言引擎阻塞 `/start` 接口

**现象**：讨论开始后只有主持人开场白，无后续专家发言。原因是 `/start` 接口使用 `await` 等待 `speaker_engine.start_discussion()` 完成，而该方法内部是无限循环，导致接口永不返回。

**解决**：改为 `asyncio.create_task(speaker_engine.start_discussion(discussion_id))`，让发言循环在后台异步运行。

**启示**：AI 生成异步代码时，容易混淆“等待任务完成”和“启动后台任务”的区别，需要明确指定。

---

### 问题 3：`supplement` / `rebuttal` 类型缺少 `reply_to_id`

**现象**：SQLite `CHECK` 约束失败，发言插入报错。因为 `speech_type` 为 `supplement` 或 `rebuttal` 时，`reply_to_id` 必须有值。

**解决**：在 `_speak` 方法中增加逻辑：当 `speech_type` 为这两种类型时，从历史发言中随机选择一条作为 `reply_to_id`。

**启示**：数据库约束是保护数据完整性的重要手段，AI 生成的代码必须与 Schema 保持同步。

---

### 问题 4：用户提问跑题

**现象**：用户输入“Xiaomi SU7 这辆车怎么样？”，但专家回答仍在讨论“智能驾驶安全验证”等宏观议题。

**解决**（部分）：将 `_build_answer_prompt` 中的用户问题提到最前面，并明确要求“完全忽略讨论话题和历史发言”。

**待完善**：`generate_speech`（自动发言）的 Prompt 仍然依赖 `discussion_topic` 和 `history_text`，导致整体讨论方向容易发散。后续可通过在创建讨论时直接将用户问题设为 `discussion_topic`，或修改 `generate_speech` 的 Prompt 来优化。

## 三、对“工程化 AI 开发”的理解

工程化 AI 开发的核心是将 AI 视作一名可交互的团队成员，通过规范化输入（结构化 Prompt、明确的上下文、分阶段任务）引导其输出可集成、可测试、可维护的成果。

### 关键原则

1. **拆解原子化**：将大任务拆分为可独立验证的子任务，每个子任务有清晰的输入/输出和验收标准。本次开发中，每个 Prompt 只聚焦一个视图（数据建模、API 设计、UI 组件、测试用例），避免 AI 一次性产出大量不可控代码。

2. **测试先行**：用测试约束 AI 产出，防止过度抽象或偏离需求。`TranscriptRepository` 的 `sequence` 自增逻辑就是通过 TDD 方式验证的。

3. **可追溯性**：每次与 AI 的交互都记录意图和修正过程（见 `PROMPTS.md`），形成“决策日志”，便于复盘和知识沉淀。

4. **人工质检闭环**：每阶段产出必须经过交叉字段映射、业务规则验证和静态数据演练。本次开发中，人工质检发现了 2 个阻塞性问题（`sequence` 缺失、`reply_to_id` 缺失）和 1 个架构问题（异步任务启动方式），均通过修正 Prompt 在下一轮迭代中解决。

### 实践总结

本次开发中，AI 负责代码生成，人工负责架构决策、质检和方向校准。这种“人工把关 + AI 执行”的模式能有效避免大模型幻觉，产出符合生产标准的代码。同时，Prompt 的质量直接影响 AI 产出质量——越清晰、越结构化的 Prompt，AI 的产出越精准。

### 后续优化方向

1. **发言聚焦性**：优化 `generate_speech` 的 Prompt，使自动发言更聚焦于用户核心问题。
2. **专家状态同步**：修复前端 `status_update` 事件接收，让专家状态灯（待机/输入中/发言中）实时变化。
3. **共识/分歧前端显示**：确保 `consensus_update` 和 `divergence_update` 事件被正确解析并渲染。
4. **部署上线**：将项目打包部署到服务器，供实际使用。
